With the network segmented, the next step was standing up the core infrastructure on pavilion. V1's Proxmox install was wiped entirely. It had no VLAN awareness, leftover VM configs from the security lab, and no clean foundation to build on. V2 started fresh.

This post covers the Proxmox reinstall, a few post-install housekeeping steps worth knowing about, and deploying Pi-hole v6, which has some meaningful differences from v5 that caught me off guard.

---

## Proxmox VE — Clean Install

### Why Reinstall

The V1 Proxmox install ran on a flat network with no VLAN configuration. Migrating an existing Proxmox node to a segmented network is possible but messy. The cleaner path was a full wipe and reinstall, configured correctly from the start with a static IP on VLAN 20 and the bridge set up to match the new network layout.

### Installation

Proxmox VE was installed from the official ISO using the same USB process from V1. During the installer, the management IP was set statically to the pavilion address on VLAN 20. The FQDN was set to `pavilion.lan` to match the OpenWRT domain.

After the install, the network configuration in `/etc/network/interfaces` reflects the bridge setup used for both host management and VM/container networking:

```
auto vmbr0
iface vmbr0 inet static
    address 192.168.20.XX/24
    gateway 192.168.20.1
    bridge-ports enp3s0
    bridge-stp off
    bridge-fd 0
```

All VMs and containers attach to `vmbr0`, which bridges to the physical ethernet interface. From the network's perspective, each workload appears as its own device on VLAN 20.

### Disabling the Enterprise Repository

Proxmox ships with the enterprise repository enabled by default, which requires a paid subscription. Without a subscription key, every `apt update` throws an authentication error. Two commands fix this:

```bash
# Disable the enterprise repo
echo "# disabled" > /etc/apt/sources.list.d/pve-enterprise.list

# Enable the community (no-subscription) repo
echo "deb http://download.proxmox.com/debian/pve bookworm pve-no-subscription" \
  > /etc/apt/sources.list.d/pve-no-subscription.list

apt update && apt dist-upgrade -y
```

The community repo is fully functional for homelab use. The enterprise repo exists for organizations that need long-term support guarantees.

### Removing the Subscription Nag

The Proxmox web UI shows a subscription nag dialog on every login when no subscription is active. One sed command removes it:

```bash
sed -i "s/data.status !== 'Active'/false/" \
  /usr/share/javascript/proxmox-widget-toolkit/proxmoxlib.js
systemctl restart pveproxy
```

Worth doing once and forgetting about.

### Access via Reverse Proxy

Rather than accessing Proxmox directly at `https://192.168.20.XX:8006`, it is proxied through Nginx Proxy Manager at `https://proxmox.jamcre.dev`. The local DNS record in Pi-hole resolves that hostname to NPM's IP, and NPM forwards the request to Proxmox with a valid wildcard SSL certificate.

One known quirk: Proxmox behind a reverse proxy shows a brief **401 No ticket** error on page load. This is a known Proxmox behavior caused by its ticket-based authentication system. Clicking OK dismisses it and the UI loads normally. Direct access at `https://192.168.20.XX:8006` is always available as a fallback without this behavior.

## Pi-hole v6

### What Changed from V1

Pi-hole v6 is a significant rewrite from v5, not just an incremental update. Two changes affected setup immediately:

- The config file changed from `setupVars.conf` to `pihole.toml`. Any v5 documentation or guides referencing `setupVars.conf` do not apply.
- The CLI command `pihole restartdns` no longer exists. Use `pihole reloaddns` instead.

The web interface looks similar but the underlying behavior is different enough that v5 guides can send you in the wrong direction.

### Container Setup

Pi-hole runs as an LXC container on pavilion with one required setting beyond the defaults: LXC nesting must be enabled. Pi-hole's FTL process does not run correctly inside a container without it.

| Setting | Value |
| :------ | :---- |
| CT ID | 100 |
| Base image | Ubuntu 22.04 (LXC template) |
| VLAN | 20 — Lab |
| CPU | 1 core |
| RAM | 512 MB |
| Disk | 8 GB |
| Nesting | Enabled |

Installation uses the same one-liner as V1:

```bash
curl -sSL https://install.pi-hole.net | bash
```

Upstream DNS was set to Cloudflare (1.1.1.1) during installation. The default StevenBlack blocklist was kept.

### Cross-VLAN DNS

This is where V2 introduced a problem that V1 never had. Pi-hole v6 defaults to listening mode `LOCAL`, which only accepts DNS queries from the same subnet the container is on. VLAN 20. Trusted devices on VLAN 10 sending queries to Pi-hole got rejected with:

```
ignoring query from non-local network
```

The fix is straightforward but easy to miss if you do not know to look for it. In the Pi-hole admin UI, navigate to Settings > DNS > Interface settings and set the listening mode to **Permit all origins**.

This is safe in this context because Pi-hole is on a firewalled internal network and port 53 is not exposed externally. The firewall ensures only trusted internal traffic can reach it.

For anyone who prefers the config file directly, the relevant setting in `/etc/pihole/pihole.toml` is:

```toml
listeningMode = "all"
```

The web UI setting handles this automatically.

### Local DNS Records

Pi-hole serves local DNS records for all lab services, mapping each subdomain to Nginx Proxy Manager's IP. NPM then routes each request to the correct backend based on the hostname.

| Hostname | Resolves To | Service |
| :------- | :---------- | :------ |
| proxmox.jamcre.dev | NPM IP | Proxmox web UI |
| pihole.jamcre.dev | NPM IP | Pi-hole admin |
| nginx.jamcre.dev | NPM IP | NPM admin panel |
| wazuh.jamcre.dev | NPM IP | Wazuh dashboard |
| nvr.jamcre.dev | NPM IP | NVR web UI |
| portainer.jamcre.dev | NPM IP | Portainer (inspiron) |
| myspeed.jamcre.dev | NPM IP | MySpeed dashboard |
| homarr.jamcre.dev | NPM IP | Homarr dashboard |

All eight records resolve to the same NPM IP. The hostname in the request is what NPM uses to route to the correct backend.
