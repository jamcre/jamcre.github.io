With Pi-hole handling DNS and all lab subdomains resolving to a single IP, the next piece was something to actually sit at that IP and route incoming requests to the right backend service. That is Nginx Proxy Manager: a reverse proxy with a web UI that handles SSL termination, certificate management, and host routing without touching config files directly.

This post covers deploying NPM, generating a wildcard SSL certificate via the Cloudflare DNS challenge, and configuring proxy hosts for every lab service.

---

## What a Reverse Proxy Does

Without a reverse proxy, accessing a service means hitting a specific IP and port. `https://192.168.20.XX:8006` for Proxmox, `http://192.168.20.XX:80` for Pi-hole, and so on. This is functional but has two problems: IP and port combos are hard to remember, and there is no clean way to put a valid SSL certificate on each one.

A reverse proxy sits in front of all those services and accepts requests by hostname instead. A request for `https://proxmox.jamcre.dev` arrives at NPM, NPM matches the hostname to a configured proxy host, and forwards the request to `https://192.168.20.XX:8006` on the backend. The client only ever talks to NPM. The backend IP and port stay internal.

The SSL certificate lives on NPM, not on each individual service. One certificate, one renewal process, all services covered.

## Wildcard SSL Certificate

Rather than generating individual certificates per service, a single wildcard certificate covers every subdomain under `*.jamcre.dev`. One cert, automatic renewal, no per-service configuration.

Let's Encrypt issues wildcard certificates only via DNS challenge, which requires proving ownership of the domain by creating a DNS TXT record. Cloudflare manages the `jamcre.dev` domain, and NPM has a built-in Cloudflare DNS provider integration that handles this automatically.

### Steps

**1. Create a Cloudflare API token**

In the Cloudflare dashboard, create an API token with Edit zone DNS permissions scoped to `jamcre.dev`. This is the minimum permission needed. Do not use the global API key.

**2. Request the certificate in NPM**

In NPM, navigate to SSL Certificates > Add SSL Certificate > Let's Encrypt and fill in:

| Field | Value |
| :---- | :---- |
| Domain | `*.jamcre.dev` |
| DNS Provider | Cloudflare |
| Credentials | `dns_cloudflare_api_token=<token>` |
| Propagation seconds | 0 |

**3. The admin email gotcha**

The NPM helper script sets the default admin account email to `admin@lab.local`. Let's Encrypt requires a valid email address for account registration and rejects `admin@lab.local`. The certificate request will fail silently if this is not updated first.

Update the NPM admin account email to a real address before requesting the certificate.

Once issued, NPM renews the certificate automatically before expiry. No manual intervention needed.

## Proxy Hosts

With the certificate in place, each service gets a proxy host entry. All hosts have Force SSL, HSTS, and HTTP/2 enabled. HSTS subdomains is off.

| Source | Backend | Notes |
| :----- | :------ | :---- |
| `proxmox.jamcre.dev` | `https://192.168.20.XX:8006` | Websockets enabled |
| `pihole.jamcre.dev` | `http://192.168.20.XX:80` | |
| `nginx.jamcre.dev` | `http://192.168.20.XX:81` | NPM proxying itself |
| `wazuh.jamcre.dev` | `https://192.168.20.XX:443` | proxy_ssl_verify off, websockets on |
| `nvr.jamcre.dev` | `http://192.168.30.XX:80` | Firewall exception required |
| `portainer.jamcre.dev` | `https://192.168.20.XX:9443` | Docker web UI on inspiron |
| `myspeed.jamcre.dev` | `http://192.168.20.XX:5216` | |
| `homarr.jamcre.dev` | `http://192.168.20.XX:7575` | |

A few entries worth explaining:

**Wazuh** uses `proxy_ssl_verify off` because the Wazuh backend serves its own self-signed certificate. NPM cannot verify it, so verification is disabled at the proxy level. The connection between the client and NPM is still fully encrypted via the wildcard cert.

**NVR** requires a firewall exception to work. The NVR sits on VLAN 30 (IoT) and NPM sits on VLAN 20 (Lab). The firewall blocks Lab to IoT traffic by default. A specific rule allows NPM's container IP to reach the NVR on TCP port 80. Config only via proxy, live camera streams are not supported through a reverse proxy.

**NPM proxying itself** at `nginx.jamcre.dev` sounds circular but works fine. The request hits NPM, which matches the hostname and forwards it to its own admin port on localhost. Useful for accessing the admin panel via a clean HTTPS URL from any device on the network.

## The Homarr/Proxmox Integration Problem

Homarr is the lab dashboard and includes a Proxmox integration that pulls resource usage data. When configured to connect directly to `https://192.168.20.XX:8006`, it fails. Proxmox at that address serves its own self-signed certificate, which Homarr cannot verify and refuses to accept.

The fix is to point Homarr at `https://proxmox.jamcre.dev` instead. That address routes through NPM, which terminates with the trusted wildcard certificate. Homarr verifies it successfully and the integration works.

This is a good illustration of why centralizing SSL termination at the proxy level has real value beyond just clean hostnames.
