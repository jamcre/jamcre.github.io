V1 was a success on its own terms. By the end of it I had a working hypervisor, a DNS sinkhole serving the whole network, and a functional attacker-target lab where I ran brute force exercises against DVWA. But the longer I used it, the more the gaps became obvious.

Everything ran on a single flat network (`192.168.1.x`), shared with every other device in the house. My homelab server sat on the same subnet as smart TVs, phones, and whatever else was connected. There was no segmentation, no isolation, and no way to enforce any kind of trust boundary between devices. If something on the IoT side misbehaved, there was nothing stopping it from reaching the lab.

Worse, there was no centralized logging. I was running attacks against myself in V1 and had no mechanism to detect them. No SIEM, no alerting, no visibility into what was happening at the endpoint level. The offensive work was meaningful but incomplete. Without detection, you're only doing half the job.

V2 was built to close those gaps deliberately.

## Designing the Conventions First

Before deploying anything, I wrote a conventions document: a reference for how IPs, container IDs, and VM IDs would be assigned. This sounds like overhead but it pays off immediately: when you're six services deep, knowing what any IP means at a glance saves real time.

### IP Convention

The same structure applies to every VLAN. Low numbers are high importance.

| Range | Category |
| :---- | :------- |
| .1 | Gateway |
| .2 — .9 | Network infrastructure (switches, APs) |
| .10 — .19 | Physical nodes (servers, workstations) |
| .20 — .49 | LXC containers |
| .50 — .74 | Virtual machines |
| .100 — .254 | DHCP pool |

So `192.168.20.11` is immediately readable as a physical node on the Lab VLAN. `192.168.20.21` is a container. `192.168.20.51` is a VM. No guessing.

### Proxmox CT/VM ID Convention

The same logic applies to Proxmox workload IDs. The first digit encodes the type.

| ID Range | Type |
| :------- | :--- |
| 1XX | LXC containers (Pi-hole, NPM, dashboards) |
| 2XX | Security VMs (Wazuh, SOC tooling) |
| 3XX | Application VMs |
| 4XX | Infrastructure VMs |
| 5XX | Dev/Lab VMs (Kali, targets) |
| 9XX | Templates |

### VLAN Design

Three VLANs with clearly defined trust levels:

| VLAN | Name | Subnet | Purpose |
| :--- | :--- | :----- | :------ |
| 10 | Trusted | 192.168.10.0/24 | Personal devices |
| 20 | Lab | 192.168.20.0/24 | Homelab infrastructure |
| 30 | IoT | 192.168.30.0/24 | Smart devices and NVR |

The firewall policy is asymmetric by design:

| Source | Destination | Policy |
| :----- | :---------- | :----- |
| Trusted | Lab | ✅ Allow |
| Trusted | IoT | ❌ Deny |
| Lab | Trusted | ❌ Deny |
| IoT | Internal | ❌ Deny |
| IoT | Internet | ✅ Allow |

Trusted can reach Lab, since I need to SSH into lab machines from my workstation. Lab cannot reach Trusted, since a compromised lab VM shouldn't be able to pivot to personal devices. IoT is fully isolated from everything internal, internet access only.

Writing this out before touching any configuration made the OpenWRT and switch setup significantly more straightforward. The policy was already defined. The implementation was just translating it.
