With the lab built and the network mapped, it was time to start working through DVWA's modules. First up: authentication brute forcing. The goal was to crack the login form at two different security levels using different tooling, and to understand what the defenses are actually doing — and why they fall short.

---

## Brute Force — Low Security (Burp Suite)

At Low security, DVWA's login form has no rate limiting, no lockout, and no token protection. Every request is independent and the response time is consistent. It's the most permissive configuration and the easiest to attack.

The approach: intercept the login POST request with **Burp Suite**, hand it to Intruder, and iterate through a wordlist against the password field.

### Intercepting the Request

With Burp's proxy active and FoxyProxy routing browser traffic through it, logging into DVWA with any credentials captures the raw POST request.

![Login POST intercepted in Burp Suite](/assets/img/posts/2025-09-15-homelab-v1-5/BRUTE-FORCE-intercept-login.png){: w="700" h="400" }_Login request captured in Burp Suite_

From there, the request is sent to Intruder, the password parameter is marked as the injection point, and a wordlist is loaded. Burp iterates through every entry, submitting a login attempt for each one.

### Result

Sorting the responses by length immediately surfaces the successful login — it returns a different page size than a failed attempt.

![Successful login rendered in browser](/assets/img/posts/2025-09-15-homelab-v1-5/BRUTE-FORCE-html-render-of-successful-login.png){: w="700" h="400" }_Admin account cracked — successful login confirmed_

The `admin` account was cracked without any friction. At Low security that's the expected outcome — the value is in understanding the workflow, not being challenged by the defenses.

## Brute Force — Medium Security (Hydra)

Medium security adds a 2-second server-side delay between login responses. The intent is to slow brute forcing to the point of impracticality. In reality, it changes the tool requirements more than the outcome.

For this round I switched to **Hydra**, which handles HTTP form-based attacks well and lets you tune timing and thread count to work around artificial delays.

### Preparing the Attack

A couple of things had to be sorted before running Hydra:

- Extracted the POST URL, form parameters, and session cookie — DVWA's medium mode requires a valid session token alongside the login attempt
- Built focused username and password lists from SecLists, filtering out entries containing apostrophes

That last point turned out to matter more than expected — more on that below.

### Running Hydra

![Hydra configured against the DVWA login form](/assets/img/posts/2025-09-15-homelab-v1-5/BRUTE-FORCE-hydra-commands.png){: w="700" h="400" }_Hydra command targeting the DVWA login form_

![Hydra successfully cracking credentials](/assets/img/posts/2025-09-15-homelab-v1-5/BRUTE-FORCE-hydra-successful-attack.png){: w="700" h="400" }_Valid credentials found for admin and pablo_

Both the `admin` and `pablo` accounts were cracked despite the 2-second delay. The core problem with this defense is that a delay slows the attack rate but doesn't cap or terminate it. Without account lockout or exponential backoff, an attacker just waits — or uses tooling that handles it automatically.

## A Discovery Outside the Scope

While manually testing login inputs, I noticed something unrelated to the brute force module: submitting a username containing an apostrophe returned a raw **SQL error message** from the application.

![Failed login exposing a SQL error](/assets/img/posts/2025-09-15-homelab-v1-5/BRUTE-FORCE-html-render-of-unsuccessful-login.png){: w="700" h="400" }_SQL error surfaced by the login form_

This is a textbook sign of string concatenation in the query rather than parameterized inputs. The database exception is leaking directly to the user, which does two things: confirms that SQL injection is likely viable here, and hands an attacker a map of the underlying query structure. It's also why the apostrophe entries were filtered out of the Hydra wordlist — they were causing error responses that muddied the attack results before I understood what was happening.

SQLi is the next DVWA module on the list.

## Defensive Takeaways

Running these attacks makes the standard hardening recommendations concrete rather than abstract:

**A time delay is not account lockout.** Medium security's 2-second delay changed the tool I used but not the outcome. Real protection means locking the account or introducing exponential backoff after a defined number of failures — something the attacker can't simply wait through.

**MFA eliminates credential brute forcing as a viable attack.** No wordlist defeats a TOTP code. Even weak passwords become irrelevant once a second factor is required.

**Never surface raw database errors to users.** The SQL exception was discovered passively, before I'd even started looking for injection surfaces. Generic error messages and parameterized queries close the vulnerability and remove the signal that reveals it.

**Authentication anomalies are detectable.** Many failed logins from a single IP, or distributed attempts across many accounts, are patterns a SIEM can alert on. Setting up detection for exactly this kind of behavior is something I built into V2 — custom Wazuh rules that fire on repeated authentication failures and surface them as alerts.

## Wrapping Up V1

V1 started with a dusty AIO pulled out of storage. By the end it was running a Type 1 hypervisor, a production network service, and a functional attacker-target lab with documented offensive exercises and defensive mitigations.
