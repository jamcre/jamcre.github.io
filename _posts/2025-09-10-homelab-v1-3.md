With Proxmox running, the next step was deploying the first real service on it. I chose **Pi-hole** — a network-wide DNS sinkhole that blocks ads and trackers before they ever reach your devices. This post covers how DNS filtering works, why I ran Pi-hole in an LXC container instead of a VM, and getting it live on the network.

---

## How Pi-hole Works

To understand Pi-hole, it helps to understand DNS first. When you type a URL into your browser, your device sends a DNS query to resolve that domain name into an IP address before it can make a connection. This happens for every domain your devices contact — websites, apps, smart home devices, all of it.

Pi-hole sits in the middle of that process. It acts as the DNS server for your network, receiving every query before it goes anywhere. When a query matches a known ad or tracker domain, Pi-hole returns nothing — the connection never happens. Everything else resolves normally.

The result is network-wide blocking that works on every device automatically, including devices where you can't install a browser extension (smart TVs, phones, IoT devices). It also gives you a live query log showing exactly what every device on your network is trying to reach, which is useful from a visibility standpoint beyond just blocking ads.

From a security perspective, DNS visibility is underrated. A lot of malware, telemetry, and C2 traffic is DNS-based. Having a log of every outbound DNS query is a simple but meaningful layer of network monitoring.

## Why an LXC Container?

Proxmox supports two types of workloads: full virtual machines and LXC containers. Pi-hole doesn't need a full VM — it's a lightweight service that runs fine on a shared kernel.

**LXC containers** share the host's kernel rather than emulating hardware. They're faster to start, use significantly less RAM, and have minimal overhead. The tradeoff is slightly less isolation than a full VM — the container shares kernel space with the host. For something like Pi-hole that doesn't run untrusted code and doesn't need a fully isolated OS, that tradeoff is fine.

Running Pi-hole in a container also makes lifecycle management easy — snapshots, backups, and rebuilds are quick operations in the Proxmox UI.

## Deployment

### Creating the Container

Using the Proxmox web UI, I created a new LXC container using an Ubuntu Server template as the base image.

| Setting | Value |
| ------- | ----- |
| Base Image | Ubuntu Server (LXC template) |
| Networking | Bridged mode (LAN visibility) |
| CPU / RAM | 1 core / 512 MB (per Pi-hole docs recommendation) |

Bridged networking is important here — the container needs to be visible on the LAN so devices can use it as their DNS server. An isolated network would defeat the purpose.

### Installing Pi-hole

With the container running, I logged into the console and ran the Pi-hole install script:

```bash
curl -sSL https://install.pi-hole.net | bash
```

The script handles everything — dependencies, service configuration, and the web interface. It prompts for an upstream DNS provider (what Pi-hole forwards non-blocked queries to — I used Cloudflare's 1.1.1.1) and which network interface to listen on. A few minutes later it's done.

![Pi-hole console installation](/assets/img/posts/2025-09-10-homelab-v1-3/pihole-console-installation.png){: w="700" h="400" }_Pi-hole installing via the console_

The admin interface is then accessible at `http://<pihole-ip>/admin`.

![Pi-hole web interface login](/assets/img/posts/2025-09-10-homelab-v1-3/pihole-web-login.png){: w="700" h="400" }_Pi-hole admin login_

### Pointing the Network at Pi-hole

Installing Pi-hole doesn't do anything on its own — devices have to actually use it as their DNS server. There are two ways to handle this: configure each device individually, or update the router so it advertises Pi-hole's IP as the DNS server via DHCP. The router approach means every device on the network gets it automatically.

Before making that change, the dashboard showed no activity — nothing was querying Pi-hole yet:

![Pi-hole dashboard with no traffic](/assets/img/posts/2025-09-10-homelab-v1-3/pihole-dashboard-router-not-pointed.png){: w="700" h="400" }_No queries — router not yet pointing to Pi-hole_

I set a static IP reservation for the container first (so the address doesn't change), then updated the router's DNS setting to Pi-hole's IP. Traffic started flowing immediately.

![Pi-hole dashboard with live queries](/assets/img/posts/2025-09-10-homelab-v1-3/pihole-dashboard-live.png){: w="700" h="400" }_Pi-hole live — query log active, blocklist working_

The query log filled up and the default blocklists began doing their job. Watching the live log is a good way to understand just how much background DNS traffic devices generate constantly — most of it goes completely unnoticed without something like this in place.
