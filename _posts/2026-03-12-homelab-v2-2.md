The first real project in V2 was the one I was least looking forward to: tearing apart the home network and rebuilding it from scratch. Everything in V1 ran on a flat `192.168.1.x` subnet shared with every device in the house. The goal for V2 was a fully segmented layout with isolated trust zones enforced at the firewall level. This post covers how that got built, what broke along the way, and what the final configuration looks like.

---

## Hardware

Two devices handle all the routing and switching:

| Device | Model | Role |
| :----- | :---- | :--- |
| Router | TP-Link Archer A7 running OpenWRT | Gateway, VLAN routing, DHCP, firewall |
| Switch | NETGEAR GS308Ev4 | 8-port managed switch, VLAN tagging |

The Archer A7 is a consumer router flashed with OpenWRT, which replaces the stock firmware with a Linux-based OS. This unlocks full VLAN support, custom firewall rules, and granular interface control that the stock firmware does not expose.

The GS308Ev4 is a low-cost managed switch that supports 802.1Q VLAN tagging. Without a managed switch, VLAN segmentation at the wired level is not possible. Unmanaged switches pass all traffic on a single broadcast domain regardless of what the router is configured to do.

---

## VLAN Design

Three VLANs, each with a defined purpose and trust level:

| VLAN | Name | Subnet | Purpose |
| :--- | :--- | :----- | :------ |
| 10 | Trusted | 192.168.10.0/24 | Personal devices: workstations, laptops, phones |
| 20 | Lab | 192.168.20.0/24 | Homelab infrastructure: servers, VMs, containers |
| 30 | IoT | 192.168.30.0/24 | Smart devices: NVR, cameras, appliances |

The firewall policy between them is asymmetric by design. Full details are in the Post 1 conventions writeup, but the short version: Trusted can reach Lab, nothing else crosses zone boundaries except to reach the internet.

## Physical Layout

The port diagram above shows how everything connects. The key points:

**Router LAN 1** carries tagged traffic for all three VLANs to the switch as a trunk. The switch strips the tags and delivers untagged traffic to each device on the correct VLAN.

**Router LAN 2** connects directly to the Annke NVR on VLAN 30. The NVR goes straight to the router rather than through the switch, which simplifies IoT isolation. The port delivers VLAN 30 traffic untagged and the NVR never sees any other network.

**Switch ports 5 through 8** were left on VLAN 1 as a dedicated management path. This turned out to be essential during setup, as explained below.

## OpenWRT Configuration

### The swconfig Model

The Archer A7 uses an older internal switch architecture called swconfig, as opposed to the newer DSA (Distributed Switch Architecture) model used in more recent hardware. This distinction matters because swconfig requires configuring the internal switch chip separately from the software network interfaces. Configuring only the software interfaces is not sufficient for DHCP to work correctly. Both layers need to be set up independently.

The internal switch table for the A7:

| VLAN | CPU | LAN 1 | LAN 2 | LAN 3 | LAN 4 | WAN |
| :--- | :-- | :---- | :---- | :---- | :---- | :-- |
| 1 | tagged | off | off | off | off | off |
| 2 | tagged | off | off | off | off | untagged |
| 10 | tagged | tagged | off | off | off | off |
| 20 | tagged | tagged | off | off | off | off |
| 30 | tagged | tagged | untagged | off | off | off |

The CPU port is tagged on all VLANs so OpenWRT can route between them. LAN 1 carries tagged traffic for VLANs 10, 20, and 30 to the switch. LAN 2 delivers VLAN 30 untagged directly to the NVR.

### Interfaces

Three network interfaces correspond to the three VLANs:

| Interface | Device | IP | DHCP Range | Notes |
| :-------- | :----- | :- | :--------- | :---- |
| trusted | br-trusted | 192.168.10.1/24 | .100 to .199 | Bridge: eth0.10 + 5 GHz radio |
| lab | eth0.20 | 192.168.20.1/24 | .100 to .199 | Wired only |
| iot | br-iot | 192.168.30.1/24 | .100 to .199 | Bridge: eth0.30 + 2.4 GHz radio |

The `trusted` and `iot` interfaces use bridge devices so that both the wired VLAN and the wireless radio share the same subnet. The `lab` interface is wired only and uses the VLAN device directly with no bridge needed.

### Bridge Devices and a Non-Obvious Gotcha

Bridge devices in OpenWRT are defined in `/etc/config/network`. On swconfig-based platforms like the Archer A7, bridge blocks must be written as anonymous blocks, not named ones. Named blocks are silently ignored by the netifd daemon. The bridge appears to configure correctly but never functions. No error is logged anywhere.

The correct format:

```
config device
    option name 'br-trusted'
    option type 'bridge'
    list ports 'eth0.10'

config device
    option name 'br-iot'
    option type 'bridge'
    list ports 'eth0.30'
```

Wireless radios join the bridge automatically via `option network` in `/etc/config/wireless`. They do not need to be listed as bridge ports.

### Firewall Zones

Four firewall zones: wan, trusted, lab, and iot.

| Zone | Input | Output | Forward | Forwards To |
| :--- | :---- | :----- | :------ | :---------- |
| wan | reject | accept | reject | |
| trusted | accept | accept | reject | wan, lab |
| lab | accept | accept | reject | wan |
| iot | reject | accept | reject | wan |

The IoT zone uses `input reject` to block unsolicited inbound traffic to the router. This immediately caused a problem: with input fully rejected, IoT devices could not receive DHCP leases or resolve DNS. Two explicit allow rules restore those two services while keeping everything else blocked:

```
config rule
    option name 'Allow-DHCP-IoT'
    option src 'iot'
    option proto 'udp'
    option dest_port '67'
    option target 'ACCEPT'

config rule
    option name 'Allow-DNS-IoT'
    option src 'iot'
    option proto 'tcp udp'
    option dest_port '53'
    option target 'ACCEPT'
```

These rules only affect input to the router itself, not inter-VLAN forwarding. IoT devices can get an IP and resolve DNS but cannot initiate connections to anything on the internal network.

### DNS

Pi-hole is pushed as the primary DNS server to all DHCP clients via dnsmasq, with Cloudflare as fallback:

```
dhcp-option=6,192.168.20.XX,1.1.1.1
```

This applies globally across all interfaces. IoT devices receive Pi-hole's IP via DHCP but cannot actually reach it since the firewall blocks IoT to Lab traffic. Their DNS falls back to Cloudflare automatically.

## NETGEAR Switch Configuration

The GS308Ev4 operates in Basic 802.1Q VLAN mode:

- Port 1 connects to the router as a trunk carrying VLANs 10, 20, and 30 tagged.
- Port 2 connects to inspiron and delivers VLAN 20 untagged.
- Port 3 connects to pavilion and delivers VLAN 20 untagged.
- Port 4 connects to citadel and delivers VLAN 10 untagged.
- Ports 5 through 8 remain on VLAN 1 as a management access path.

That last point was not planned. It became necessary after the switch management interface became unreachable twice during VLAN reconfiguration, requiring factory resets both times. Leaving ports 5 through 8 on VLAN 1 gives direct access to the switch management page from any device plugged into those ports, regardless of what is happening with the rest of the VLAN config. Learned that one the hard way.

## Firewall Exception: NVR Access via Reverse Proxy

One exception to the strict IoT isolation was added after initial setup. Nginx Proxy Manager needed a path to the NVR web interface to proxy access via `nvr.jamcre.dev`. The NVR sits on VLAN 30 and NPM sits on VLAN 20. By default, Lab to IoT traffic is blocked.

A specific rule was added to allow only this one connection:

| Rule | Source | Destination | Port | Reason |
| :--- | :----- | :---------- | :--- | :----- |
| Allow-NPM-to-NVR | NPM container (VLAN 20) | NVR (VLAN 30) | TCP 80 | NVR web UI via reverse proxy |

The rule is scoped to a single source IP and destination port, keeping the broader IoT isolation intact.
