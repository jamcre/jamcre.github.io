With Wazuh running and all four agents enrolled, the next step was actually using it. Wazuh ships with hundreds of built-in rules covering common attack patterns, but a core SOC skill is writing detection logic tailored to your environment: understanding which base rule IDs fire in your specific setup, knowing how to chain rules for correlation, and validating that alerts actually fire before trusting them in production.

This post covers four custom rules written, tested, and verified against live attack simulations on the lab machines.

---

## Rule Architecture

Before writing any rules, three concepts are essential to understand. Getting these wrong produces rules that appear syntactically correct but silently never fire.

### if_sid vs if_matched_sid

These two directives look similar but behave fundamentally differently.

`if_sid` fires your rule when a single event matches the parent rule ID. One event triggers one alert. This is appropriate for detections where a single occurrence is immediately suspicious, such as an unauthorized sudo attempt. A regular user does not accidentally try to run a command they do not have access to.

`if_matched_sid` fires your rule when a parent rule has matched a defined number of times within a time window. This is correlation: counting repeated events rather than reacting to one. Brute force detection requires this approach because a single failed login is normal. Thirty failed logins from the same IP in sixty seconds is not.

### frequency and timeframe Are Attributes, Not Elements

A common XML mistake is placing `frequency` and `timeframe` as child elements inside the rule body:

```xml
<!-- WRONG -->
<rule id="100001" level="10">
  <frequency>5</frequency>
  <timeframe>120</timeframe>
  ...
</rule>
```

They must be attributes on the opening `<rule>` tag:

```xml
<!-- CORRECT -->
<rule id="100001" level="10" frequency="5" timeframe="120">
  ...
</rule>
```

The wrong version produces no error. The rule loads but never fires at the threshold.

### same_srcip vs same_field

`same_srcip` groups frequency counts by source IP. It works correctly for Linux events where the decoder populates the standard `srcip` field.

For Windows events, the attacker IP is stored in `win.eventdata.ipAddress`, a dynamic decoded field. The standard `srcip` field is empty. A Windows correlation rule using `same_srcip` will never reach its frequency threshold because no two events share the same empty `srcip` value. Use `same_field` with the correct decoded field name instead:

```xml
<same_field>win.eventdata.ipAddress</same_field>
```

This is the most non-obvious failure mode in the entire exercise. The rule loads, events arrive, and nothing fires. The only way to catch it is knowing to look for this specific behavior.

## Verify Base Rule IDs First

Before writing any correlation rule, the actual base rule ID firing for each event type must be verified in the lab environment. Documentation lists expected rule IDs, but the actual firing rule depends on the Wazuh version and agent OS. Assuming a rule ID without verification leads to correlation rules that silently never fire.

Method used: generated test events (failed SSH logins, failed Windows logons, failed sudo attempts), then observed alerts in Wazuh dashboard under Threat Hunting > Events, and noted the `rule.id` on each actual alert.

| Event Type | Expected SID | Actual SID in Lab | Verified |
| :--------- | :----------- | :---------------- | :------- |
| SSH invalid user | 5710 | 5710 | Yes |
| SSH auth failure | 5716 | 5760 | Yes — 5760 fires in this environment |
| Linux sudo failure | 5401 | 5405 | Yes — 5405 fires in this environment |
| Windows Event ID 4625 | 60122 | 60122 | Yes |

Two of the four base rule IDs differed from what documentation suggested. Writing correlation rules against the wrong parent ID is a silent failure with no error message.

## The Rules

All rules live in `/var/ossec/etc/rules/local_rules.xml` on the Wazuh manager. Custom rule IDs use the 100000 to 999999 range to avoid conflicts with built-in rules.

```xml
<group name="custom,local,">

  <!-- RULE 100001: SSH Brute Force (invalid user)
       Fires when rule 5710 matches 5 times from the same source IP
       within 120 seconds. Targets attempts against non-existent
       usernames, common in automated scanning. -->
  <rule id="100001" level="10" frequency="5" timeframe="120">
    <if_matched_sid>5710</if_matched_sid>
    <same_srcip />
    <description>Custom: SSH brute force — repeated attempts against invalid users from $(srcip)</description>
    <mitre>
      <id>T1110</id>
    </mitre>
    <group>authentication_failures,ssh_brute_force,</group>
  </rule>

  <!-- RULE 100002: SSH Auth Failure Spike
       Fires when rule 5760 matches 8 times from the same source IP
       within 60 seconds. Targets rapid failures against existing
       accounts, consistent with a password spray pattern.
       Higher severity and shorter timeframe than 100001 because
       targeting real accounts indicates a more focused attack. -->
  <rule id="100002" level="12" frequency="8" timeframe="60">
    <if_matched_sid>5760</if_matched_sid>
    <same_srcip />
    <description>Custom: Aggressive SSH auth failures — possible active brute force from $(srcip)</description>
    <mitre>
      <id>T1110</id>
    </mitre>
    <group>authentication_failures,ssh_brute_force,</group>
  </rule>

  <!-- RULE 100003: Linux Unauthorized Sudo Attempt
       Fires on a single event from rule 5405. No frequency needed.
       A regular user does not accidentally attempt sudo on commands
       they lack access to. Single-event detection is appropriate
       here — one attempt warrants investigation. -->
  <rule id="100003" level="12">
    <if_sid>5405</if_sid>
    <description>Custom: Unauthorized sudo attempt — $(dstuser) is not in sudoers on $(hostname)</description>
    <mitre>
      <id>T1548.003</id>
    </mitre>
    <group>privilege_escalation,sudo_attempt,</group>
  </rule>

  <!-- RULE 100004: Windows Failed Logon Spike
       Fires when rule 60122 (Event ID 4625) matches 5 times within
       120 seconds. Uses same_field referencing win.eventdata.ipAddress
       instead of same_srcip. See notes above on why same_srcip does
       not work for Windows events. -->
  <rule id="100004" level="10" frequency="5" timeframe="120">
    <if_matched_sid>60122</if_matched_sid>
    <same_field>win.eventdata.ipAddress</same_field>
    <description>Custom: Windows failed logon spike — $(win.eventdata.ipAddress) on $(hostname)</description>
    <mitre>
      <id>T1110</id>
    </mitre>
    <group>authentication_failures,windows_brute_force,</group>
  </rule>

</group>
```

## Validation

### Syntax Check Before Deployment

`wazuh-logtest` was used to validate rule matching logic before restarting the manager. Sample SSH failure logs were pasted interactively, and the `firedtimes` counter was observed incrementing with each paste. On the fifth paste, the correlation rule fired as expected.

One note: `wazuh-logtest -t` is not a valid flag in this Wazuh version. Syntax errors are caught by restarting the manager and checking the output of `systemctl status wazuh-manager` for parse errors. Restart the manager after saving any rule changes:

```bash
systemctl restart wazuh-manager
```

### Attack Simulations

Each rule was tested with a targeted simulation designed to reliably trigger the specific event path the rule monitors.

**Rules 100001 and 100002 — SSH**

Failed SSH logins were generated from citadel targeting pavilion using a PowerShell loop:

```powershell
# Rule 100001 — invalid user path (rule 5710)
for ($i=1; $i -le 6; $i++) {
    ssh fakeuser@192.168.20.XX 2>$null
    Start-Sleep -Milliseconds 500
}
```

For rule 100002, a test user was created on the target Linux host with a known password, then rapid failures were generated against that existing account to trigger the auth failure path (rule 5760 rather than 5710).

**Rule 100003 — Sudo**

A non-privileged test user was created on a Linux agent. Logged in as that user and attempted:

```bash
sudo cat /etc/shadow
```

The PAM subsystem logged the unauthorized attempt, Wazuh decoded it via rule 5405, and custom rule 100003 fired on the single event.

**Rule 100004 — Windows**

`net use` was used to generate network logon failures against a remote share, which reliably populates `win.eventdata.ipAddress` in the Event ID 4625 log:

```powershell
for ($i=1; $i -le 6; $i++) {
    net use \\192.168.20.XX\ipc$ /user:fakeuser wrongpassword 2>$null
    Start-Sleep -Milliseconds 500
}
```

Using `Start-Process` or GUI-based credential prompts was found to be unreliable for populating `win.eventdata.ipAddress`. The `net use` method is consistent and recommended.

## Key Lessons

**Verify base rule IDs in your environment before writing correlation rules.** Two of the four expected IDs were wrong. 5760 fired instead of 5716 for SSH auth failures. 5405 fired instead of 5401 for sudo. A correlation rule written against the wrong parent ID produces no error and never fires.

**same_srcip does not work for Windows events.** Windows Event Log events store the source IP in `win.eventdata.ipAddress`, not in the standard `srcip` field. Using `same_srcip` on a Windows correlation rule means the frequency threshold is never reached. Use `same_field` with the correct decoded field name.

**frequency and timeframe are rule attributes, not child elements.** The wrong placement produces no error but the threshold logic never activates.

**Single-event rules do not need frequency or timeframe.** Rule 100003 fires on one event because one unauthorized sudo attempt is already suspicious. Adding a frequency threshold would delay detection unnecessarily and add complexity without benefit.

**Test event source matters for SSH rules.** Generating failures against non-existent users (rule 5710 path) behaves differently from generating failures against existing users with wrong passwords (rule 5760 path). Both paths need to be tested separately if both correlation rules are in scope.
