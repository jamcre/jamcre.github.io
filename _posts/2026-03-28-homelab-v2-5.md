The gap that motivated rebuilding the lab in the first place was visibility. In V1, attacks were running against lab machines with no centralized logging, no alerting, and no way to know what was actually happening at the endpoint level. Wazuh is what closes that gap.

Wazuh is an open source SIEM and endpoint detection platform. It ingests logs from enrolled agents, runs detection rules against them, and surfaces alerts in a central dashboard. This post covers deploying it as a single-node instance on pavilion and enrolling agents across all four lab machines.

Detection rules and attack simulations are covered in the next post. This one focuses on getting the platform running.

---

## What Wazuh Does

Wazuh has three main components, all running together in a single-node deployment:

- **Wazuh server** — receives logs and events from agents, runs detection rules, generates alerts
- **Wazuh indexer** — stores events and alerts (built on OpenSearch)
- **Wazuh dashboard** — web UI for querying events, viewing alerts, and managing agents

Agents are lightweight processes installed on each machine. They ship logs, file integrity events, and security telemetry to the server continuously. The server decodes them, matches them against rules, and fires alerts when thresholds are met.

For a homelab, Wazuh gives you the same detection workflow used in production SOC environments: ingest, decode, correlate, alert. Writing rules, tuning thresholds, and investigating alerts are all core SOC skills. Running them against real machines you control is a much better learning environment than a simulated lab.

## Deployment

### Why the OVA Method

Wazuh offers two primary installation methods: a quickstart script that installs everything on a fresh Linux host, and a pre-built OVA image that contains a fully configured single-node stack. The OVA was chosen here for one reason: it eliminates installation variables.

The quickstart script is sensitive to the host OS version, available memory, and network configuration. The OVA boots into a known-good state with the indexer, server, and dashboard already configured and running. For a lab where the goal is detection and rule writing rather than installation troubleshooting, the OVA is the cleaner starting point.

### VM Setup

The OVA was downloaded directly to pavilion via wget and imported as a VM disk:

```bash
wget https://packages.wazuh.com/4.x/vm/wazuh-4.x.ova
```

A VM was created in Proxmox with no OS and no storage, the OVA disk was attached, and the VM was started. The result is a fully running Wazuh stack with no OS installation step.

| Setting | Value |
| :------ | :---- |
| VM ID | 200 |
| Deployment type | Single-node (server + indexer + dashboard) |
| OS | Wazuh official OVA |
| VLAN | 20 — Lab |
| CPU | 3 cores |
| RAM | 8 GB |
| Disk | 25 GB |

3 cores and 8 GB RAM is the recommended minimum for a single-node Wazuh deployment with multiple agents. Running below this causes the indexer to become slow under load.

### Access

Wazuh is proxied through NPM at `https://wazuh.jamcre.dev`. As noted in the previous post, NPM requires `proxy_ssl_verify off` for this proxy host since Wazuh's backend serves a self-signed certificate. The direct address at `https://192.168.20.XX:443` remains available as a fallback.

## Agent Enrollment

Four machines, four different operating systems. Each gets a Wazuh agent installed and registered to the manager.

| Agent | OS | Method | Status |
| :---- | :- | :----- | :----- |
| citadel | Windows 11 | MSI installer | Active |
| pavilion | Linux (Proxmox host) | DEB package | Active |
| annex | macOS Sequoia | PKG installer | Active |
| inspiron | Ubuntu Server 24.04 | DEB package | Active |

The Windows and macOS installs follow the standard Wazuh documentation without friction: download the appropriate installer, run it with the manager IP, the agent registers and starts shipping data.

Linux installs using the DEB package had two issues worth documenting.

### Issue 1: Missing lsb-release Dependency

On minimal Debian-based systems, the `lsb-release` package is not installed by default. The Wazuh agent package installation halts if it is missing. Install it first:

```bash
apt install lsb-release -y
```

Then proceed with the Wazuh agent installation. This is not documented prominently in the official guide and will silently fail the installation if skipped.

### Issue 2: MANAGER_IP Placeholder Not Replaced

The DEB package installer does not always populate the `MANAGER_IP` value correctly in `/var/ossec/etc/ossec.conf`. In the pavilion install, the placeholder string `MANAGER_IP` was left literally in the config file rather than being replaced with the actual Wazuh server IP.

The agent fails to start entirely with an invalid config. The symptom is the agent service refusing to start with no obvious error message in the terminal. Check the log directly:

```bash
cat /var/ossec/logs/ossec.log
```

The config issue is logged there. Fix it by editing ossec.conf, replacing `MANAGER_IP` with the actual Wazuh server IP, then registering the agent manually and restarting:

```bash
/var/ossec/bin/agent-auth -m 192.168.20.XX
systemctl restart wazuh-agent
```

After that the agent registers, connects, and begins shipping data.

## Where Things Stand

Wazuh is running on VLAN 20 with agents enrolled on all four lab machines. Events from Windows, Linux, and macOS are all flowing into the indexer and visible in the dashboard. The detection infrastructure is in place.
