With the Pavilion upgraded and ready, it was time to figure out what to actually do with it. The answer was virtualization — specifically, turning it into a bare-metal hypervisor using **Proxmox VE**. This post covers what that means, why it was the right choice, and an installation that didn't go entirely smoothly.

---

## What is a Hypervisor?

A hypervisor is software that lets one physical machine run multiple isolated virtual machines simultaneously. Each VM gets its own allocated CPU, RAM, and storage and behaves like an independent computer — from the network's perspective it looks like a separate machine, and from the host's perspective it's just a process consuming resources.

There are two types worth knowing:

**Type 2 hypervisors** run on top of an existing operating system. VirtualBox and VMware Workstation are the common examples — you install them on Windows or macOS like any other application and run VMs inside that. This works fine for casual use, but the host OS is always running underneath, consuming resources and adding a layer of overhead between your VMs and the actual hardware.

**Type 1 hypervisors** run directly on bare metal. There's no host OS — the hypervisor *is* the operating system. Better performance, better resource efficiency, and the machine is purpose-built for running virtual workloads rather than doing it as a side task.

For a dedicated homelab server, Type 1 is the right choice.

## Why Proxmox?

The main free options for a Type 1 hypervisor are Proxmox VE and VMware ESXi. ESXi's free tier has become increasingly restricted since Broadcom's acquisition, making Proxmox the practical choice for homelabs. It also wins on its own merits:

- **Web-based management UI** — everything is managed from a browser, no separate client needed
- **Supports both VMs and LXC containers** — more flexibility than hypervisors limited to full VMs
- **Active community** — extensive documentation, forums, and community scripts for common tasks
- **Genuinely free and open source** — no license expiration or feature gating

The enterprise subscription unlocks certain update channels, but the free tier is fully functional. It's also common enough in small production environments that learning it has real-world relevance beyond the homelab.

## Installation

### Preparing the Installation Media

The process starts with downloading the Proxmox VE ISO from the official site and flashing it to a USB drive. I used **balenaEtcher** for this — select the ISO, select the drive, flash.

![Flashing the Proxmox ISO with balenaEtcher](/assets/img/posts/2025-09-06-homelab-v1-2/flash-proxmox-iso-balenaetcher.png){: w="700" h="400" }_Flashing the Proxmox ISO_

Straightforward in theory. The Pavilion had other ideas.

### The Legacy Boot Problem

With the USB ready, I entered the BIOS, disabled every other boot source, and pointed it at the USB. The installer wouldn't load — the drive simply wasn't being detected.

![USB drive not detected during boot](/assets/img/posts/2025-09-06-homelab-v1-2/drive-not-detected.jpg){: w="700" h="400" }_Drive not detected_

After some research, the fix was enabling **Legacy Boot** in the BIOS. Modern systems default to UEFI-only, but the Pavilion is old enough that this causes conflicts with certain bootloaders. Enabling Legacy Boot alongside UEFI let the system see the drive.

![Enabling Legacy Boot in BIOS](/assets/img/posts/2025-09-06-homelab-v1-2/enable-legacy-boot.jpg){: w="700" h="400" }_Enabling Legacy Boot_

I also switched from balenaEtcher to **Ventoy** — a tip picked up from others hitting the same wall. Ventoy turns a USB into a bootable launcher that reads ISOs dropped directly onto it, no reflashing needed. It resolved the detection issue and is genuinely useful to keep around — one drive, multiple ISOs, boot any of them.

![Ventoy bootloader menu](/assets/img/posts/2025-09-06-homelab-v1-2/ventoy-bootloader.jpg){: w="700" h="400" }_Ventoy bootloader_

### Running the Installer

With the USB recognized, the Proxmox installer loaded cleanly.

![Proxmox VE installer](/assets/img/posts/2025-09-06-homelab-v1-2/proxmox-installer.jpg){: w="700" h="400" }_Proxmox VE installer_

The installer walks through a handful of steps:

- Select the target disk — the 1 TB SSD installed in Post 1
- Set a root password and admin email
- Configure the hostname and management IP

The management IP is worth being deliberate about — it's the address used to access the web UI going forward, so it should be stable. I set a static assignment on the router to keep it from shifting after a reboot.

### First Login

After rebooting, Proxmox VE was accessible from any browser on the network at `https://<proxmox-ip>:8006`.

![Proxmox web interface login screen](/assets/img/posts/2025-09-06-homelab-v1-2/proxmox-web-interface-login.png){: w="700" h="400" }_Proxmox login screen_

![Proxmox VE dashboard](/assets/img/posts/2025-09-06-homelab-v1-2/proxmox-web-interface-home.png){: w="700" h="400" }_Proxmox VE dashboard_

The dashboard shows the node, available resources, and storage. Seeing a proper server management interface running on a machine that was collecting dust a week earlier was satisfying. The Pavilion was officially a hypervisor.
