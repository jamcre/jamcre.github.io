Pavilion handles all the core infrastructure: Proxmox, Pi-hole, NPM, and Wazuh. The Inspiron was brought in as a second node dedicated to Docker workloads, keeping those services off the primary host and leaving pavilion's resources available for the heavier VM workloads.

This post covers preparing the Inspiron as a Ubuntu Server Docker host and deploying three services on it: Portainer for container management, Homarr as a unified dashboard, and MySpeed for network speed tracking.

---

## The Inspiron as a Docker Host

The Dell Inspiron 3505 was repurposed from a personal laptop to a dedicated server. The decision to run Ubuntu Server directly rather than installing Proxmox was intentional: Proxmox experience was already covered through pavilion, and adding a hypervisor layer would consume resources without any benefit for Docker-only workloads.

### Hardware

| Spec | Value |
| :--- | :---- |
| CPU | AMD Ryzen 5 3450U (4c/8t) |
| RAM | 12 GB DDR4 |
| Storage | 256 GB NVMe + 1 TB HDD |
| OS | Ubuntu Server 24.04.4 LTS |
| VLAN | 20 — Lab |
| Network | USB Gigabit adapter (primary) |

One hardware limitation worth calling out: the built-in ethernet port tops out at 100 Mbps. For a server handling file transfers and container traffic, that is a meaningful bottleneck. A USB 3.0 Gigabit adapter was added and configured as the primary network interface for lab use.

### Post-Install Configuration

A few settings were applied after the base Ubuntu Server install before anything else was deployed.

**Timezone:**

```bash
timedatectl set-timezone America/New_York
```

**Root SSH login disabled** in `/etc/ssh/sshd_config`:

```
PermitRootLogin no
```

**UFW enabled with SSH allowed:**

```bash
ufw allow ssh
ufw enable
```

**Lid close sleep disabled:**

Running a laptop as a headless server means the lid will be closed. By default, Ubuntu suspends when the lid closes. Prevent this by editing `/etc/systemd/logind.conf` and setting all lid switch handlers to ignore:

```
HandleLidSwitch=ignore
HandleLidSwitchExternalPower=ignore
HandleLidSwitchDocked=ignore
```

Then reload the service:

```bash
systemctl restart systemd-logind
```

Without this, the machine suspends the moment the lid goes down and becomes unreachable.

**Docker installed** via the official install script, with the lab user added to the docker group to avoid requiring sudo for every Docker command:

```bash
usermod -aG docker jamcre
```

SSH access from citadel:

```bash
ssh jamcre@192.168.20.XX
```

## Portainer

Portainer is a web-based management UI for Docker. Rather than managing containers entirely through the CLI, Portainer provides a dashboard for viewing running containers, pulling images, checking logs, and managing volumes and networks.

It runs as a Docker container itself and is accessible via NPM at `https://portainer.jamcre.dev`.

```bash
docker run -d \
  --name portainer \
  --restart=always \
  -p 9443:9443 \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v portainer_data:/data \
  portainer/portainer-ce:latest
```

Portainer listens on port 9443 over HTTPS using its own self-signed certificate. NPM proxies it at `https://portainer.jamcre.dev` with `proxy_ssl_verify off` for the same reason as Wazuh: the backend serves a self-signed cert that NPM cannot verify.

## Homarr

Homarr is a homelab dashboard that provides a unified interface for accessing and monitoring all lab services. Rather than bookmarking individual IP and port combinations, Homarr presents everything as a grid of service tiles with live status indicators.

It was installed on pavilion as an LXC container using the Proxmox VE Helper Scripts, then accessed via NPM at `https://homarr.jamcre.dev`.

Two integrations were configured:

**Pi-hole integration** connects via API token and pulls live query statistics into the Homarr dashboard. Configured through the Pi-hole admin panel under Settings > API to generate a token.

**Proxmox integration** connects via an API token with the PVEAuditor role and pulls node resource usage. One specific requirement: Homarr must connect to Proxmox via `https://proxmox.jamcre.dev` rather than directly to `https://192.168.20.XX:8006`. Connecting directly fails because Proxmox serves a self-signed certificate that Homarr cannot verify. Routing through NPM resolves this since NPM terminates the connection with the trusted wildcard certificate.

## MySpeed

MySpeed is a self-hosted network speed testing tool that runs scheduled speed tests and maintains a history of results over time. It gives a long-term view of ISP performance rather than a single snapshot from a speed test site.

Deployed as an LXC container on pavilion using the Proxmox VE Helper Scripts, accessible via NPM at `https://myspeed.jamcre.dev`.

No configuration beyond the defaults was needed. Once deployed and proxied, it begins running tests on a schedule and logs results to an internal database.

## Wrapping Up V2

With the secondary node running and the service layer complete, V2 was fully operational:

- Three-VLAN network with enforced trust boundaries
- Proxmox hosting all core infrastructure workloads
- Pi-hole handling cross-VLAN DNS with local hostname resolution
- Nginx Proxy Manager routing all services to clean HTTPS hostnames
- Wazuh providing centralized log ingestion and endpoint detection across all four machines
- Custom detection rules covering SSH brute force, Windows logon spikes, and unauthorized privilege escalation
- A Docker host running Portainer, Homarr, and MySpeed as supporting services

V2 ran as the production lab environment until a hardware upgrade made a fresh V3 build the better path forward. The next series covers what changed and why.
